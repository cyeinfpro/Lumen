# 通用服饰电商模特展示图工作流方案

> 状态：MVP 闭环已落地；V2/V3 能力作为后续扩展保留
> 适用范围：Lumen 的图片工作流扩展功能
> 用户决策：通用服饰；优先模特一致性、商品还原度和高级质感；采用方案 A，先确认模特本人，再让模特试穿商品。

## 1. 背景与目标

当前 Lumen 已具备通用文生图、图生图、视觉问答和长任务能力。服饰电商图生成不是简单的“一张商品图生成一张模特图”，它需要先稳定模特形象，再稳定商品还原，最后稳定电商出片质量。

本功能的目标是增加一个面向服饰商品图的结构化图片工作流：

1. 用户上传衣服商品图，并用自然语言描述期望。
2. 系统先生成可确认的模特方案，包括三视图和大头照。
3. 用户确认模特、妆发、饰品和整体搭配方向。
4. 系统再把确认后的模特方案与商品图结合，生成电商模特展示图。
5. 系统对成图做自动质检，用户可以局部返修或重生成。

这不是替代通用 `text_to_image` / `image_to_image`，而是在它们之上增加一个“分阶段、有确认点、有质量门槛”的工作流层。

## 2. 设计原则

### 2.1 先定人，再穿衣

采用方案 A：第一阶段只确认模特本人，不让模特提前试穿商品。这样可以把“模特人设不满意”和“商品融合不满意”分开处理，降低返工成本。

第一阶段重点：

- 脸部稳定
- 身材比例稳定
- 发型、妆容、气质稳定
- 三视图之间的一致性稳定

第二阶段才处理：

- 商品穿着效果
- 衣服版型、颜色、图案、材质还原
- 场景、姿态、光线和电商质感

### 2.2 质量优先

用户明确选择“模特一致性、商品还原度和高级质感”优先，因此默认策略不是最快、最省，而是：

- 候选模特少而精，优先可控一致性。
- 最终图允许多步生成和质检重试。
- 关键图使用较高质量配置。
- 失败时明确指出问题，而不是把低质量图当成成功结果。

### 2.3 通用服饰优先

第一版面向通用服饰，而不是只服务单一女装或男装。系统需要支持：

- 上衣：T 恤、衬衫、卫衣、针织衫、夹克等
- 下装：裤子、半裙、长裙等
- 连体类：连衣裙、连体裤等
- 外套：大衣、风衣、羽绒服、西装等
- 成套搭配：套装、运动套装等
- 配饰型商品：围巾、帽子、包、腰带等

MVP 可以先聚焦成人服饰。童装、内衣、泳装等更敏感或更依赖姿态规范的品类，应作为后续受控扩展。

## 3. 核心用户旅程

### 3.1 发起工作流

用户从「项目」进入，点击「新建项目」，选择「服饰模特展示图」模板。系统打开一个项目新建页，而不是打开聊天对话。

项目新建页只要求用户完成三件事：

1. 上传 1 到 3 张商品图。
2. 填写一句基础需求，例如“高级通勤感，冷淡气质模特，适合独立站女装”。
3. 点击「创建项目并开始分析」。

系统创建一个项目，并把商品图、基础需求和后续所有阶段状态绑定到该项目。后续推进通过项目详情页的阶段按钮完成，而不是通过连续对话完成。

### 3.2 商品理解

系统先分析商品图，给出结构化摘要：

- 商品品类
- 主色
- 材质
- 版型
- 领口、袖型、长度、腰线等关键结构
- 图案、logo、纽扣、拉链、口袋等细节
- 商品图质量风险，如遮挡、低清、色偏、只有局部图

用户可以直接确认，也可以修正：

- “这是米白色，不是纯白。”
- “面料是亚麻，不是棉。”
- “logo 必须保留。”

### 3.3 生成模特候选

系统生成 3 套模特候选。每套候选包含：

- 正面全身
- 侧面全身
- 背面全身
- 大头照
- 模特设定摘要
- 推荐妆发
- 推荐饰品方向
- 适合的拍摄风格

三视图和大头照用于确认“同一个模特”的稳定性。此阶段不穿商品，只生成模特本人和基础服装或中性贴身底装，避免商品还原问题干扰模特确认。

### 3.4 用户确认方案

用户可以执行以下操作：

- 选择一个完整模特方案
- 锁定某个方案的人脸
- 锁定某个方案的身材比例
- 更换发型、妆容、肤色光感、表情风格
- 选择或关闭饰品方案
- 要求重生成某一套候选

MVP 先支持“选择一个完整模特方案”和“一次文字微调”。更细的脸、身材、发型拆分可以放在第二阶段。

### 3.5 生成电商展示图

用户确认后，系统生成一组结果图。默认建议输出 4 张：

1. 主图：干净背景，全身正面，突出商品轮廓。
2. 姿态图：自然站姿或轻微动态，展示穿着氛围。
3. 细节图：半身或局部，突出面料、领口、袖口、图案等。
4. 侧背图：侧面或背面角度，展示版型和长度。

用户也可以选择模板：

- 白底电商图
- 高级灰棚拍
- 城市通勤场景
- 生活方式场景
- 社媒种草图

### 3.6 质检与返修

系统对结果做自动质检，重点检查：

- 商品颜色是否偏差过大
- 版型是否被改造
- 图案、logo、纽扣、口袋是否丢失或多出来
- 模特脸是否和已确认方案一致
- 手、腿、脸、衣服边缘是否有明显生成错误
- 电商图是否清晰、干净、有高级质感

如果质检失败，系统不直接把结果标成完成，而是提示：

- “第 2 张商品领口被改窄，建议重生成。”
- “第 4 张背面图和确认模特脸部一致性无关，可保留。”
- “第 1 张颜色偏暖，建议按商品原图校色。”

用户可以选择局部返修、整张重生成或保留结果。

## 4. 信息架构与界面设计

### 4.1 一级导航调整

由于后续不止“服饰模特展示图”一个结构化工作流，建议把一级导航从：

```text
创作 / 图库 / 我的
```

调整为：

```text
创作 / 项目 / 图库 / 我的
```

“项目”承载所有有阶段、有状态、可恢复、可继续编辑的结构化工作流。“工作流”是系统实现语言，不建议作为用户可见一级入口；“项目”更符合用户心智，用户会认为自己是在做一组图、一批商品、一个任务，而不是在管理流程。

四个一级入口的边界如下：

| 入口 | 负责什么 | 不负责什么 |
|---|---|---|
| 创作 | 快速发起聊天、生图、图生图、选择工作流模板 | 不管理长周期任务列表 |
| 项目 | 管理结构化工作流、草稿、进行中任务、历史项目、批量任务 | 不做普通图片瀑布流 |
| 图库 | 管理所有图片资产、筛选、查看、下载、复用图片 | 不展示复杂阶段状态 |
| 我的 | 账号、偏好、系统设置、模特库、品牌默认风格 | 不承载日常创作主流程 |

“服饰模特展示图”在信息架构中的路径：

```text
创作
  -> 选择“服饰模特展示图”
  -> 创建项目
  -> 进入项目详情页

项目
  -> 查看所有服饰模特展示项目
  -> 继续未完成项目
  -> 复用历史项目

图库
  -> 查看项目产出的商品图、模特候选图、最终展示图
```

推荐路由：

```text
/projects
/projects/new
/projects/apparel-model-showcase/new
/projects/:projectId
```

如果未来有更多工作流，可以继续复用同一项目体系：

- 商品图精修项目
- 批量商品上新项目
- 海报套图项目
- 社媒种草图项目
- 品牌视觉项目

### 4.2 项目页结构

“项目”页不是图库，也不是聊天页。它应该优先回答三个问题：

1. 哪些任务正在进行？
2. 哪些项目需要我确认？
3. 哪些项目已经完成，可以继续复用或下载？

项目首页建议分区：

- 进行中：正在生成、排队中、质检中
- 待确认：商品理解待确认、模特方案待确认、返修建议待确认
- 最近完成：已完成项目，可打开结果或复用配置
- 项目模板：服饰模特展示图、商品图精修、批量上新等

项目卡片应展示：

- 项目类型
- 商品缩略图或代表图
- 当前阶段
- 状态
- 最近更新时间
- 产出数量
- 下一步动作，如“确认模特”“查看质检”“继续生成”

### 4.3 服饰模特展示项目详情

服饰模特展示图进入项目详情页后，页面主体围绕阶段推进，而不是围绕聊天气泡。

推荐布局：

- 顶部：项目名、状态、保存时间、下载入口
- 左侧：阶段导航
- 中间：当前阶段工作区
- 右侧：商品信息、已确认模特、关键约束、质检摘要

移动端可折叠为：

- 顶部阶段条
- 当前阶段内容
- 底部固定主操作按钮
- 商品信息和约束放入抽屉

从「创作」进入时，系统创建一个项目；从「项目」进入时，继续已有项目。这样既保留创作入口的轻量感，也让复杂任务有自己的管理空间。

### 4.4 项目流程生产台

该功能的主界面应设计为“项目流程生产台”，不是对话式聊天界面。

核心原则：

- 每个阶段都有固定表单、固定状态和明确主按钮。
- 用户通过「确认」「生成」「返修」「下一步」推进流程。
- AI 说明和用户补充需求可以存在，但只能作为辅助区域，不是主路径。
- 项目可以暂停、恢复、回看和复用。
- 每一步的输入、输出、确认记录都沉淀在项目中。

推荐三栏布局：

```text
左侧：步骤导航
中间：当前步骤操作区
右侧：项目资产与约束
```

左侧步骤导航：

```text
1 上传商品
2 商品理解
3 模特设定
4 模特候选
5 方案确认
6 商品融合
7 质检返修
8 交付
```

中间操作区根据当前阶段变化：

- 上传商品：上传框、商品缩略图、基础需求输入
- 商品理解：结构化商品信息、风险提示、修正表单
- 模特设定：人群、气质、妆发、身材比例、禁用项
- 模特候选：3 套模特方案对比
- 方案确认：选中模特、饰品方案、最终确认
- 商品融合：出片模板、张数、比例、质量模式
- 质检返修：质检报告、问题图、返修说明
- 交付：最终图、下载、入图库、保存模特

右侧项目资产与约束：

- 商品原图
- 商品必须保留的细节
- 已确认模特
- 饰品方案
- 输出规格
- 质检摘要

聊天框不作为主界面。若保留 AI 助手，只能放在右下角或辅助抽屉，用于：

- 解释当前阶段
- 帮用户润色需求
- 填写返修说明
- 回答“为什么这张图需要返修”

用户不应该依赖聊天来推进流程。

### 4.5 工作流阶段

前端应明确展示阶段，而不是只显示普通聊天气泡。

建议阶段条：

```text
上传商品 -> 商品理解 -> 模特设定 -> 模特候选 -> 方案确认 -> 商品融合 -> 质检返修 -> 交付
```

每个阶段都应该有独立状态：

- `waiting_input`
- `running`
- `needs_review`
- `approved`
- `failed`
- `completed`

### 4.6 工作区布局

建议使用独立项目详情页，不在普通会话流里塞工作流卡片。项目详情页展示当前阶段的核心内容。

模特方案阶段：

- 顶部：商品缩略图和用户需求摘要
- 中部：3 套模特候选卡片
- 底部：确认按钮、微调说明框、重生成按钮

每套候选卡片内容：

- 四宫格图片：正面、侧面、背面、大头照
- 简短标签：气质、年龄感、身材比例、发型
- 操作：选择、微调、重生成本套

商品融合阶段：

- 左侧：已确认模特参考
- 中部：商品图和关键还原点
- 右侧：输出模板选择
- 底部：生成按钮和质量偏好说明

结果阶段：

- 网格展示最终图
- 每张图显示质检状态
- 支持下载、放大、从此图继续编辑、局部返修

### 4.7 交互细节

用户确认动作必须显式：

- “确认此模特”
- “确认商品信息”
- “开始生成展示图”

高成本生成前应显示预计消耗：

- 预计生成张数
- 是否使用高质量模式
- 是否生成 4K 终稿
- 预计等待时间范围

不要让用户误以为模特方案图就是最终商品图。模特方案卡片需要标注为“未试穿商品，仅用于确认模特形象”。

## 5. 生成策略

### 5.1 商品理解

商品理解可以使用视觉问答能力完成，输出结构化 JSON：

```json
{
  "category": "blazer",
  "color": "warm ivory",
  "material_guess": "structured woven fabric",
  "silhouette": "relaxed straight fit",
  "key_details": ["notched lapel", "single button", "front flap pockets"],
  "must_preserve": ["color", "lapel shape", "button position", "pocket placement"],
  "risks": ["front-only product image", "fabric texture not very clear"]
}
```

用户修正信息后，系统把最终商品摘要作为后续生成的硬约束。

### 5.2 模特候选生成

模特候选生成不使用商品试穿图。输入包括：

- 用户对目标人群和风格的要求
- 商品品类和调性摘要
- 平台风格，如淘宝、天猫、独立站、Instagram、小红书
- 品牌关键词，如高级、极简、甜酷、运动、通勤、户外

输出要求：

- 每套候选是同一个合成模特
- 正面、侧面、背面、大头照保持同一身份
- 使用中性基础服装或贴身基础装，不能抢商品风格
- 不出现真实名人或可识别公众人物
- 不生成与用户商品相似的服装细节，避免用户误判最终试穿效果

### 5.3 模特身份锁定

用户确认模特后，系统保存以下参考：

- 大头照，作为脸部身份参考
- 正面全身，作为身材比例参考
- 三视图，作为体态和发型参考
- 模特描述文本，作为提示词约束

后续商品融合时，不只依赖文字描述，而要同时使用已确认图片作为 reference。

### 5.4 商品融合生成

商品融合阶段输入：

- 商品原图
- 商品结构化摘要
- 已确认模特参考图
- 饰品和妆发选择
- 目标输出模板
- 输出张数和比例

融合提示词必须把优先级写清楚：

1. 商品真实性优先。
2. 已确认模特一致性优先。
3. 电商图质感优先。
4. 场景和姿态服务于商品展示。

提示词应明确禁止：

- 改变商品颜色
- 改变领口、袖口、衣长、腰线等结构
- 擅自添加图案、logo、口袋、腰带
- 过度褶皱导致版型变形
- 用外套、头发、手臂遮挡商品关键区域

### 5.5 推荐输出规格

模特方案图：

- 3 套候选
- 每套 1 张四宫格图或 4 张独立图
- 优先 3:4 或 4:5
- 使用中高质量，避免低清脸部导致用户误判

最终展示图：

- MVP 默认 4 张
- 电商主图建议 3:4 或 4:5
- 场景图可支持 4:5、3:4、16:9
- 终稿默认 2K，高质量模式
- 对重要主图提供 4K 终稿选项

## 6. 数据模型建议

可以在现有 `conversations/messages/tasks/images` 之上增加工作流表，不建议把所有状态塞进普通消息。

### 6.1 workflow_runs

记录一次完整工作流。

关键字段：

- `id`
- `conversation_id`
- `user_id`
- `type`，固定为 `apparel_model_showcase`
- `status`
- `product_image_ids`
- `current_step`
- `quality_mode`，如 `premium`
- `created_at`
- `updated_at`

### 6.2 workflow_steps

记录每个阶段的状态和输入输出。

关键字段：

- `id`
- `workflow_run_id`
- `step_key`
- `status`
- `input_json`
- `output_json`
- `task_ids`
- `image_ids`
- `approved_at`
- `approved_by`

`step_key` 示例：

- `product_analysis`
- `model_candidates`
- `model_approval`
- `look_assembly`
- `showcase_generation`
- `quality_review`

### 6.3 model_candidates

记录模特候选，方便后续复用。

关键字段：

- `id`
- `workflow_run_id`
- `candidate_index`
- `portrait_image_id`
- `front_image_id`
- `side_image_id`
- `back_image_id`
- `contact_sheet_image_id`
- `model_brief_json`
- `status`
- `selected_at`

### 6.4 quality_reports

记录自动质检结果。

关键字段：

- `id`
- `workflow_run_id`
- `image_id`
- `overall_score`
- `product_fidelity_score`
- `model_consistency_score`
- `aesthetic_score`
- `artifact_score`
- `issues_json`
- `recommendation`

## 7. API 设计草案

## 8. Prompt 合约

### 8.1 商品理解 Prompt

用途：从商品图抽取稳定约束。

要点：

- 只描述图中可见信息。
- 不确定的地方标记为 `unknown`。
- 把必须保留的商品细节单独列出。
- 输出结构化 JSON，便于后续生成和质检复用。

### 8.2 模特候选 Prompt

用途：生成三视图和大头照。

模板要点：

```text
Create a synthetic fashion model concept sheet for ecommerce apparel usage.
The model is not wearing the user's product yet.
Generate one consistent synthetic model across front view, side view, back view, and close-up headshot.
Use simple neutral base clothing only.
Keep identity, face, hairstyle, body proportion, and skin lighting consistent across all views.
Style direction: {style_prompt}.
Product category context: {product_category}, used only to choose compatible model styling.
Do not copy any real person, celebrity, influencer, or identifiable private individual.
Do not add product-specific garment details at this stage.
```

### 8.3 商品融合 Prompt

用途：让确认模特试穿商品。

模板要点：

```text
Create a premium ecommerce model showcase image.
Use the confirmed synthetic model reference for face, hairstyle, body proportion, and overall identity.
Use the product image as the locked garment reference.
Preserve the garment color, silhouette, neckline, sleeve shape, length, fabric impression, pattern, logo, button, pocket, and seam placement.
The model should wear the product naturally and realistically.
The pose and styling must support clear product display.
Do not change the garment design.
Do not add new patterns, logos, pockets, belts, or decorative elements.
Do not cover important garment details with arms, hair, accessories, or props.
Shot type: {shot_type}.
Scene style: {scene_style}.
Output should look like a high-end ecommerce campaign image, clean, sharp, realistic, and commercially usable.
```

### 8.4 质检 Prompt

用途：对成图进行视觉复核。

检查项：

- 商品是否仍是同一件商品
- 商品颜色是否接近原图
- 结构细节是否保留
- 模特是否接近已确认候选
- 图像是否有明显生成瑕疵
- 是否适合作为电商图

输出示例：

```json
{
  "overall_score": 82,
  "product_fidelity_score": 78,
  "model_consistency_score": 88,
  "aesthetic_score": 86,
  "artifact_score": 80,
  "issues": [
    {
      "severity": "medium",
      "type": "product_fidelity",
      "message": "The neckline appears slightly narrower than the product reference."
    }
  ],
  "recommendation": "revise"
}
```

## 9. 质量标准

### 9.1 商品还原度

合格标准：

- 主色接近商品图。
- 关键版型没有明显变化。
- 图案、logo、纽扣、口袋、拉链等关键细节没有缺失或新增。
- 商品没有被手臂、头发、包、外套严重遮挡。

强制返修：

- 商品颜色明显变成另一种颜色。
- 领口、袖型、衣长、裤型等结构明显变化。
- logo 或图案错乱。
- 商品变成了另一件衣服。

### 9.2 模特一致性

合格标准：

- 大头照与最终图脸部身份接近。
- 发型、脸型、妆容、年龄感一致。
- 身材比例和体态基本一致。

强制返修：

- 最终图明显变成另一个人。
- 同一组结果图中模特身份不一致。
- 脸部严重变形。

### 9.3 高级质感

合格标准：

- 光线干净，主体清晰。
- 背景不抢商品。
- 姿态自然，符合服饰电商展示。
- 没有廉价棚拍感、过度磨皮、过度锐化或强滤镜。

强制返修：

- 手、脚、脸、衣服边缘明显错误。
- 背景杂乱或主体不突出。
- 商品展示信息不足。

## 10. MVP 范围

第一版建议只做最小可用闭环：

1. 支持上传 1 到 3 张商品图。
2. 支持商品理解摘要和用户修正。
3. 生成 3 套模特候选。
4. 每套候选包含三视图和大头照。
5. 用户选择一个完整模特方案。
6. 用户选择一个饰品方案，或关闭饰品。
7. 生成 4 张电商展示图。
8. 每张图生成自动质检报告。
9. 支持一次文字返修。
10. 支持下载最终图。

MVP 暂不做：

- 多商品批量上新
- 多模特同场景
- 精细图层编辑
- 自动抠图换背景工作台
- 人脸、身材、发型自由拼装
- 童装、内衣、泳装等特殊品类的完整支持
- 品牌级长期模特库

## 12. 风险与对策

### 12.1 商品细节被改

风险：生成模型容易把衣服重设计。

对策：

- 商品理解阶段抽取 `must_preserve`。
- 商品融合 Prompt 中把商品真实性列为最高优先级。
- 质检阶段对比商品图和结果图。
- 对主图支持局部返修和重生成。

### 12.2 模特不一致

风险：三视图、大头照和最终图不是同一个人。

对策：

- 模特候选阶段生成 concept sheet。
- 确认后保存多张 reference。
- 最终生成时同时传大头照和全身参考。
- 每组最终图做模特一致性评分。

### 12.3 用户把草稿图当最终图

风险：模特候选图不穿商品，用户可能误解。

对策：

- UI 明确标注“未试穿商品，仅用于确认模特形象”。
- 工作流阶段条明确显示当前处于“模特方案”阶段。
- 确认按钮文案使用“确认此模特”，不用“生成完成”。

### 12.4 通用品类过宽

风险：通用服饰范围大，不同品类规则差异明显。

对策：

- 商品理解阶段先分类。
- 根据品类选择不同 shot plan。
- MVP 限制在成人日常服饰。
- 特殊品类进入受控模板。

### 12.5 成本和等待时间较高

风险：质量优先会带来更多生成和质检调用。

对策：

- 候选阶段默认 3 套，不无限生成。
- 最终默认 4 张。
- 4K 只对用户选中的主图生成。
- 高成本动作前显示预计消耗。

## 13. 验收标准

功能可上线的最低标准：

1. 用户能从一张商品图启动完整工作流。
2. 系统能生成 3 套可选择的模特候选。
3. 每套候选都包含正面、侧面、背面和大头照。
4. 用户能确认一个模特方案。
5. 系统能基于确认模特和商品图生成 4 张展示图。
6. 最终图中商品不能明显变成另一件衣服。
7. 同一组最终图中的模特身份应基本一致。
8. 每张最终图都有质检结论。
9. 用户能对至少一张图发起文字返修。
10. 工作流关窗后可恢复，任务状态不丢失。

## 14. 推荐实施顺序

第一阶段：产品闭环

1. 增加工作流状态模型。
2. 接入商品理解。
3. 生成模特候选。
4. 完成模特确认 UI。
5. 生成最终展示图。

第二阶段：质量门槛

1. 增加自动质检。
2. 增加失败原因展示。
3. 增加一次文字返修。
4. 增加结果下载。

第三阶段：效率和复用

1. 保存模特库。
2. 保存品牌默认风格。
3. 增加搭配方案模板。
4. 支持批量商品。

## 15. 结论

该功能应被设计成“服饰图片工作流”，而不是单个图片生成按钮。关键产品判断是把不确定性拆开：先确认模特，再确认搭配，最后生成商品展示图。这样可以同时提高模特一致性、商品还原度和最终高级质感，也符合用户对质量优先的要求。
